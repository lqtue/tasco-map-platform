## Quyết định chiến lược

### 1. OSM là lớp dữ liệu chuẩn
Toàn bộ thuộc tính bản đồ công cộng — giới hạn tốc độ, đèn tín hiệu, biển cấm, phân loại đường — đều được đẩy trực tiếp lên OpenStreetMap. Không dùng database thuộc tính riêng. Valhalla routing engine đọc OSM trực tiếp → dữ liệu OSM sạch = routing tốt hơn mà không cần thêm công.

### 2. Yandex Maps là tài liệu tham chiếu UX
V-Base-Maps và V-Navigation-Maps clone UX của Yandex Maps / Yandex Navigation sát nhất có thể. Thiện chịu trách nhiệm xây dựng Figma design system với toàn bộ user flow.

### 3. Mapbox API là tài liệu tham chiếu API
Toàn bộ tên gọi, cấu trúc URL, tham số API (geocoding, routing, search) đều theo chuẩn Mapbox API. Lý do: thống nhất ngôn ngữ trong team + dễ viết tài liệu sau này.

### 4. TomTom cho lớp hiển thị giao thông thời gian thực
TomTom dùng nền đường OSM → Way ID khớp hoàn toàn với bản đồ nền của TASCO. Dùng TomTom traffic tile API làm lớp overlay trên mobile app và web map. Chỉ để hiển thị — không dùng để tính ETA.

### 5. Công việc ảnh vệ tinh lùi lại trong T6
Tile server ảnh vệ tinh của Tuệ tạm hoãn. Ảnh trước mắt: đặt mua qua UP42 hoặc Skywatch.com. Đang tuyển thêm remote sensing scientist để cùng phụ trách từ T9/T10 khi vệ tinh 80 cm của đối tác Nga phóng.

### 6. Chiến lược speed limit: gộp 3 lớp
Ba nguồn được overlay để xác định giới hạn tốc độ cho từng đoạn đường OSM:
1. **Mặc định theo luật** — luật giao thông Việt Nam ánh xạ sang phân loại đường OSM (Quân)
2. **Nhận diện biển báo thực tế** — Mapillary API + Google Street View API (Tuệ)
3. **Đối chiếu với app khác** — so sánh với các app dẫn đường hiện có

Kết quả sau gộp → đẩy lên OSM dưới dạng tag `speed_limit` / `maxspeed`.

### 7. Kiến trúc geocoding: Amazon Location + TomTom
- Chính: Amazon Location Service (provider: HERE) — reverse geocode tâm H3 → lấy địa chỉ dạng text cấu trúc
- Phụ: TomTom reverse geocoding — bổ sung
- Dữ liệu đầu vào: Google Open Buildings → cluster theo H3 resolution 11 → lấy tâm mỗi ô
- Một lần gọi API trả về ~15–20 địa chỉ lân cận
